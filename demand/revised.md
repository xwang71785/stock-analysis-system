# 2. Revised Analysis and Decision Framework

## 2.2.2 Document Impact Analysis Module

### Objective
Move beyond generic sentiment analysis and determine whether a news item, filing, macro release, or KOL post has **material impact** on a target stock for a **next-week swing trading horizon**.

### Supported Document Types
- Financial news
- Exchange announcements / company filings
- Macro and policy releases
- KOL / crowd posts

### Core Functions
For each document, the system should automatically extract:

1. **Target relevance**
   - Is this document relevant to a stock in the tracked universe?
   - Relevance type:
     - Direct
     - Indirect
     - Sector-level
     - Market-wide

2. **Impact direction**
   - Positive
   - Negative
   - Mixed
   - Neutral

3. **Impact strength**
   - Weak
   - Medium
   - Strong

4. **Impact horizon**
   - Intraday
   - 2–5 trading days
   - 1–2 weeks

5. **Event type**
   Examples:
   - Earnings / pre-announcement
   - Guidance change
   - Major contract/order
   - Product launch / approval
   - Policy support / policy restriction
   - Financing / private placement / convertible bond
   - Shareholder increase / reduction
   - Management change
   - Litigation / regulation
   - Supply chain / customer / competitor development

6. **Confidence**
   - Low
   - Medium
   - High

7. **Evidence extraction**
   - Key sentence(s) supporting the classification
   - Structured explanation generated by LLM

### Processing Approach
Use a hybrid pipeline:
- Rule-based preprocessing:
  - deduplication
  - source normalization
  - stock/entity keyword matching
- Cloud LLM classification:
  - impact relevance
  - event type
  - direction
  - horizon
  - confidence
  - explanation

### Output
For each document-stock pair, store:
- relevance score
- impact direction
- impact strength
- impact horizon
- event type
- confidence
- explanation
- evidence snippet

### Notes
- This module replaces simple positive/negative sentiment as the primary text-analysis mechanism.
- Generic sentiment may still be retained as a supplementary feature but should not be the main decision variable.

## 2.2.4 Order Book and Intraday Pressure Analysis Module

### Objective
Analyze intraday order-book and tick behavior to assess **buying/selling pressure**, **execution quality**, and **entry/exit timing confirmation** for next-week swing trades.

### Positioning
This module should be treated as:
- **microstructure pressure proxy**
- **timing confirmation layer**
- **risk/confirmation signal**

It should **not** be described as definitive “institution behavior analysis,” since five-level order-book data alone is insufficient to identify institutional intent with high confidence.

### Inputs
- Real-time five-level order book
- Historical tick data
- Intraday trade/quote snapshots
- Daily OHLCV data

### Recommended Features

#### 1. Order Imbalance
- Sum of bid volume vs ask volume
- Rolling imbalance over 1-min / 5-min / 15-min windows

#### 2. Large Order Pressure
- Large bid/ask appearance frequency
- Net large-order pressure
- Persistence of large orders over time

#### 3. Cancellation Proxy
- Estimate order cancellation intensity by comparing order book changes over time
- Identify rapid add/cancel behavior
- Detect abnormal quote instability

#### 4. VWAP and Intraday Strength
- Current price vs VWAP
- Intraday close strength
- Afternoon strength vs morning weakness
- Intraday recovery / fade patterns

#### 5. Liquidity and Price Response
- Spread changes
- Depth changes
- Price response after large quote/trade events
- Abnormal liquidity vacuum behavior

### Derived Signals
- Buying pressure strengthening
- Selling pressure strengthening
- Breakout confirmation
- Failed breakout warning
- Distribution risk
- Weak close warning
- Intraday absorption / support behavior

### Output
The module should produce:
- pressure score
- confirmation score
- crowding risk flag
- explanation of dominant intraday behavior

## 2.2.5 Swing Trading Scoring and Recommendation System

### Objective
Generate an explainable, next-week oriented trading assessment for each stock in the tracked universe.

### Output Structure
Instead of a single raw score only, the system should generate three core scores plus one recommendation layer:

1. **Opportunity Score (0–100)**
   - Measures upside attractiveness for the next week

2. **Risk Score (0–100)**
   - Measures probability of setup failure, reversal, crowding, or event risk

3. **Confidence Score (0–100)**
   - Measures consistency, data quality, and reliability of the current signal

4. **Trade Recommendation**
   - Strong Long
   - Watch Long
   - Neutral
   - Avoid
   - Reduce / Exit

### Core Signal Dimensions

#### A. Event / Filing Impact
Weight: **30%**
Includes:
- exchange announcements
- earnings-related events
- policy impact
- major contracts/orders
- capital market actions
- material company-specific developments

Role:
- Primary driver for next-week swing opportunities

#### B. Price / Trend Structure
Weight: **25%**
Includes:
- MA trend alignment
- breakout / breakdown state
- 20-day / 60-day price structure
- relative strength vs sector / index
- ATR-normalized trend quality

Role:
- Confirms whether the market is validating the thesis

#### C. Volume / Participation
Weight: **15%**
Includes:
- volume surprise
- turnover percentile
- price-volume confirmation
- accumulation/distribution proxy

Role:
- Measures participation quality behind the move

#### D. Order Book / Intraday Pressure
Weight: **15%**
Includes:
- order imbalance
- large-order pressure
- cancellation proxy
- VWAP strength
- close strength

Role:
- Confirms entry timing and identifies short-term risk

#### E. Sector / Macro Regime
Weight: **10%**
Includes:
- sector leadership / weakness
- market regime
- policy and macro context
- thematic tailwind or headwind

Role:
- Adjusts stock score based on external environment

#### F. Crowd Attention / KOL Proxy
Weight: **5%**
Includes:
- KOL mention spike
- crowd sentiment
- narrative acceleration
- crowding risk

Role:
- Used mainly as attention/crowding indicator, not standalone alpha source

---

### Opportunity Score Calculation
The Opportunity Score should emphasize:
- positive material events
- bullish price confirmation
- strong relative strength
- supportive volume
- positive intraday pressure
- favorable sector backdrop

### Risk Score Calculation
The Risk Score should emphasize:
- contradictory or negative events
- failed breakout behavior
- weak closing structure
- abnormal crowding spike
- high cancellation/unstable order-book behavior
- adverse macro/policy developments

### Confidence Score Calculation
The Confidence Score should be based on:
- agreement across signal modules
- source reliability
- LLM impact classification confidence
- data freshness and completeness
- consistency between narrative and price action

### Recommendation Logic
Recommended initial rule set:

- **Strong Long**
  - High Opportunity
  - Low to moderate Risk
  - Medium to high Confidence
  - Positive event/thesis support
  - Price structure confirms

- **Watch Long**
  - Positive Opportunity
  - Acceptable Risk
  - Requires confirmation
  - Example: wait for breakout or pullback entry

- **Neutral**
  - Mixed or weak signals
  - No clear edge

- **Avoid**
  - Low Opportunity
  - Elevated Risk
  - Weak or contradictory confirmation

- **Reduce / Exit**
  - Negative event impact
  - Thesis deterioration
  - Failed price structure
  - Rising crowding / reversal risk

### Explainability Requirement
For every generated recommendation, the system must display:
- top 3 bullish drivers
- top 3 bearish drivers
- what changed since prior day
- confidence explanation
- supporting documents / evidence
- suggested entry condition
- suggested invalidation / stop logic
- optional target style or exit condition

### Suggested UI Presentation
For each stock:
- Opportunity Score
- Risk Score
- Confidence Score
- Trade Recommendation
- Entry Condition
- Invalidation Level
- Top Positive Drivers
- Top Negative Drivers
- Latest Material Events

# 3. Revised Technical Architecture

## 3.1 Technology Stack

| Layer | Technology | Description |
|------|------------|-------------|
| Backend | Python 3.11+ | Core development language |
| Web Framework | FastAPI | High-performance API and service layer |
| Task Scheduling | APScheduler (v1) / Celery or Prefect (v2+) | Data collection and processing workflows |
| Relational Storage | PostgreSQL | Primary operational database for app, metadata, documents, signals, alerts |
| Analytical Storage | DuckDB + Parquet | Historical market data, tick-derived features, backtesting, analytical queries |
| Frontend | HTML + CSS + JS (v1) | Lightweight local web interface |
| Real-time Updates | WebSocket / SSE | Push score and alert updates |
| LLM Integration | Cloud LLM API | Relevance, event extraction, impact assessment, explanation generation |
| Data Processing | Pandas / Polars | Feature engineering and transformation |
| Logging / Monitoring | Structured logging + health checks | Job status, parser errors, stale data detection |

### Architecture Principles
1. **Operational and analytical workloads should be separated**
   - PostgreSQL for transactional app state
   - DuckDB/Parquet for time-series analytics

2. **Text analysis should be event-impact driven**
   - not generic sentiment only

3. **Every score should be explainable**
   - score provenance must be stored

4. **Real-time data should be aggregated into useful features**
   - avoid storing only raw snapshots without derived features

# 4. Revised Database Schema

## 4.1 Design Principles
1. Separate **documents**, **features**, **scores**, and **recommendations**
2. Store **LLM outputs explicitly**
3. Store **explanations and evidence**
4. Use **time-series analytical storage** for high-frequency data
5. Ensure stock-level uniqueness constraints are correct

---

## 4.2 Core Relational Tables (PostgreSQL)

### Stocks
```sql
CREATE TABLE stocks (
    id SERIAL PRIMARY KEY,
    code VARCHAR(10) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    industry VARCHAR(100),
    theme VARCHAR(100),
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### Documents
Unified table for news, filings, macro, and KOL posts.

```sql
CREATE TABLE documents (
    id BIGSERIAL PRIMARY KEY,
    source_type VARCHAR(20) NOT NULL,   -- news, filing, macro, kol
    source_name VARCHAR(50) NOT NULL,   -- sina, eastmoney, cninfo, weibo, etc.
    title TEXT,
    url TEXT UNIQUE,
    content TEXT,
    author_name VARCHAR(100),
    published_at TIMESTAMP,
    collected_at TIMESTAMP DEFAULT NOW(),
    content_hash VARCHAR(64),
    language VARCHAR(10) DEFAULT 'zh',
    raw_metadata JSONB
);

CREATE INDEX idx_documents_source_type ON documents(source_type);
CREATE INDEX idx_documents_published_at ON documents(published_at);
CREATE INDEX idx_documents_content_hash ON documents(content_hash);
```

### Document-Stock Impact Mapping
Stores structured LLM/rule outputs.

```sql
CREATE TABLE document_stock_impacts (
    id BIGSERIAL PRIMARY KEY,
    document_id BIGINT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    relevance_score REAL NOT NULL,              -- 0 to 1
    relevance_type VARCHAR(20),                 -- direct, indirect, sector, market
    impact_direction VARCHAR(20),               -- positive, negative, mixed, neutral
    impact_strength VARCHAR(20),                -- weak, medium, strong
    impact_horizon VARCHAR(20),                 -- intraday, 2_5d, 1_2w
    event_type VARCHAR(50),
    confidence VARCHAR(20),                     -- low, medium, high
    explanation TEXT,
    evidence_snippet TEXT,
    model_name VARCHAR(100),
    model_version VARCHAR(50),
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_doc_impacts_stock_id ON document_stock_impacts(stock_id);
CREATE INDEX idx_doc_impacts_document_id ON document_stock_impacts(document_id);
CREATE INDEX idx_doc_impacts_event_type ON document_stock_impacts(event_type);
```

### Daily Features
Stores computed daily-level features for ranking/scoring.

```sql
CREATE TABLE daily_features (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    trade_date DATE NOT NULL,
    close_price REAL,
    volume REAL,
    amount REAL,

    trend_score REAL,
    relative_strength_score REAL,
    breakout_score REAL,
    volatility_score REAL,
    volume_surprise REAL,
    turnover_percentile REAL,

    event_score REAL,
    crowd_attention_score REAL,
    sector_regime_score REAL,

    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(stock_id, trade_date)
);

CREATE INDEX idx_daily_features_stock_date ON daily_features(stock_id, trade_date);
```

### Intraday / Order Book Features
Stores aggregated intraday pressure features, not just raw snapshots.

```sql
CREATE TABLE intraday_features (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    feature_time TIMESTAMP NOT NULL,

    order_imbalance REAL,
    large_order_pressure REAL,
    cancel_ratio_proxy REAL,
    vwap_distance REAL,
    close_strength REAL,
    liquidity_stress REAL,
    pressure_score REAL,

    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_intraday_features_stock_time ON intraday_features(stock_id, feature_time);
```

### Crowd Features
KOL-based crowd/attention indicators.

```sql
CREATE TABLE crowd_features (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    feature_date DATE NOT NULL,

    mention_count INT,
    attention_spike_score REAL,
    crowd_sentiment_score REAL,
    sentiment_dispersion REAL,
    crowding_risk_score REAL,

    created_at TIMESTAMP DEFAULT NOW(),
    UNIQUE(stock_id, feature_date)
);

CREATE INDEX idx_crowd_features_stock_date ON crowd_features(stock_id, feature_date);
```

### Scores
Final score outputs per stock and date/time.

```sql
CREATE TABLE scores (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    score_time TIMESTAMP NOT NULL,

    opportunity_score REAL NOT NULL,   -- 0 to 100
    risk_score REAL NOT NULL,          -- 0 to 100
    confidence_score REAL NOT NULL,    -- 0 to 100

    event_component REAL,
    trend_component REAL,
    volume_component REAL,
    order_flow_component REAL,
    regime_component REAL,
    crowd_component REAL,

    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_scores_stock_time ON scores(stock_id, score_time);
```

### Recommendations
Stores final action suggestions.

```sql
CREATE TABLE recommendations (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    recommendation_time TIMESTAMP NOT NULL,

    stance VARCHAR(20),                -- bullish, neutral, bearish
    action VARCHAR(30),                -- strong_long, watch_long, neutral, avoid, reduce_exit
    entry_condition TEXT,
    invalidation_level REAL,
    target_hint TEXT,
    confidence_label VARCHAR(20),      -- low, medium, high

    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_recommendations_stock_time ON recommendations(stock_id, recommendation_time);
```

### Score Explanations
Human-readable reasoning layer.

```sql
CREATE TABLE score_explanations (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT NOT NULL REFERENCES stocks(id) ON DELETE CASCADE,
    explanation_time TIMESTAMP NOT NULL,

    top_positive_factors JSONB,
    top_negative_factors JSONB,
    changes_from_prior JSONB,
    supporting_document_ids JSONB,
    contradiction_flags JSONB,
    narrative_summary TEXT,

    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_score_explanations_stock_time ON score_explanations(stock_id, explanation_time);
```

### Alerts
```sql
CREATE TABLE alerts (
    id BIGSERIAL PRIMARY KEY,
    stock_id INT REFERENCES stocks(id) ON DELETE CASCADE,
    alert_time TIMESTAMP NOT NULL DEFAULT NOW(),
    alert_type VARCHAR(50) NOT NULL,
    severity VARCHAR(20),              -- info, warning, critical
    title TEXT,
    message TEXT,
    is_read BOOLEAN DEFAULT FALSE,
    related_score_id BIGINT,
    related_document_id BIGINT,
    created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_alerts_stock_time ON alerts(stock_id, alert_time);
CREATE INDEX idx_alerts_is_read ON alerts(is_read);
```

### Settings
```sql
CREATE TABLE settings (
```
